2단계(도출)와 3단계(구현) 사이의 **게이트**. 빠지면 얇은 디자인이 그대로 통과한다.

원칙 1-2는 *반복 루프*가 원본을 좇는 걸 금한다(베끼기 회귀 방지). 그러나 **도출이 원본의 본질을
재해석해 담았는지 한 번 점검**하는 건 베끼기가 아니라 필수다 — 이 점검이 없어서 디자인이 얇게 새는 게
실제로 관측됐다. (구현↔design 은 4단계가 보지만, design↔reference 는 아무도 안 봐서 생긴 구멍.)

1. `npm run design-preview -- $ARGUMENTS` → `design/preview.html` 생성(색/타이포/간격/그림자 토큰 + 컴포넌트 카드 + **원본 썸네일 대조**).
2. `npm run serve -- $ARGUMENTS root 4174` → http://127.0.0.1:4174/design/preview.html (사람이 보는 디자인 게이트. 단 스펙 시트라 실제 사이트 렌더는 아님 — 그래서 아래 3을 에이전트가 한다).
3. **에이전트가 design↔reference 충실도 채점**(사람이 스펙만으론 판단 어려움): `reference/source*.png` 를 Read 하고,
   `design/*.json` 이 원본의 **밀도·프로모 에너지·기능 풍부함·타이포 펀치·실사 활용·구성요소**를 (베끼지 않고)
   재해석해 담았는지 렌즈별 0~1 채점 + 갭 목록. **자기채점 편향을 줄이려 독립 에이전트 fan-out 권장**(Workflow).
4. 게이트:
   - 충실도 < 0.8 → 갭대로 `design/` 보강(토큰 추가/컴포넌트 intent·expects 두껍게/누락 컴포넌트 추가) 후 1로.
   - 충실도 ≥ 0.8 → `/03-implement`.

핵심: 이 게이트는 "구현이 잘됐나"가 아니라 **"디자인 의도 자체가 원본을 담았나"**를 묻는다. 둘은 다른 게이트다
(도출 충실도 / 구현 수렴). 둘을 한 게이트가 겸하면 얇은 의도가 통과한다.
